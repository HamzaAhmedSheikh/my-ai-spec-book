# Implementation Plan: Docusaurus Documentation Site for Physical AI & Humanoid Robotics Book

**Branch**: `001-docusaurus-setup` | **Date**: 2025-11-29 | **Spec**: [spec.md](./spec.md)
**Input**: Feature specification from `/specs/001-docusaurus-setup/spec.md`

## Summary

Initialize a Docusaurus documentation website to present the "Physical AI & Humanoid Robotics" textbook content. The implementation extracts content from a source markdown file, distributes it across 10 structured documentation pages, and deploys a functional static site with proper navigation. This is an MVP-focused setup prioritizing speed-to-deployment over extensive customization.

**Technical Approach**: Use Docusaurus Classic preset with TypeScript, leverage autogenerated sidebars for navigation, extract content from source textbook via script-based section detection, and preserve all original formatting. Root URL redirects directly to the introduction page.

## Technical Context

**Language/Version**: Node.js >= 18.0, TypeScript (via Docusaurus --typescript flag)
**Primary Dependencies**: Docusaurus 3.x (classic preset), React (bundled with Docusaurus), npm
**Storage**: Static files (markdown content in docs/, built output in build/)
**Testing**: Manual verification via `npm start` (dev server) and `npm run build` (production build)
**Target Platform**: Static site generator output, deployable to any static hosting (GitHub Pages planned for future)
**Project Type**: Static documentation site (Docusaurus monorepo structure)
**Performance Goals**: Dev server start <30s, production build <2min, zero build errors
**Constraints**: Must preserve 100% of source formatting, zero content rewriting, content extraction must be automated
**Scale/Scope**: 10 documentation pages, single sidebar category, minimal configuration customization

## Key Design Decisions

### Decision 1: Autogenerated Sidebar vs Manual Sidebar Items

**Options Considered**:
1. **Autogenerated sidebar** (SELECTED):
   - Uses `type: 'autogenerated', dirName: 'physical-ai'`
   - Order controlled by `sidebar_position` frontmatter in each file
   - Automatically discovers new files if added later
2. Manual sidebar items:
   - Explicitly lists each document: `['physical-ai/introduction', 'physical-ai/quarter-overview', ...]`
   - Full control over order and nesting
   - Requires manual updates when adding files

**Decision**: Autogenerated sidebar

**Rationale**:
- **Scalability**: If content expands (e.g., additional sections, nested subcategories), autogenerated sidebars adapt automatically
- **Maintainability**: No need to edit `sidebars.ts` when adding/removing pages
- **Best Practice**: Docusaurus documentation recommends autogenerated for docs-heavy sites
- **Frontmatter Control**: `sidebar_position` provides explicit ordering without manual config duplication

**Trade-offs**:
- ✅ Faster iteration for content changes
- ✅ Less error-prone (no typos in file paths)
- ❌ Less explicit (must check frontmatter to understand order)

**ADR Required**: ⚠️ **YES** - This is an architecturally significant decision that affects long-term maintainability and content workflow. Document as ADR-001: "Sidebar Generation Strategy".

**Suggested ADR Title**: "ADR-001: Use Autogenerated Sidebars for Course Textbook Navigation"

---

### Decision 2: Docs at Root (`/`) vs Docs at `/docs/`

**Options Considered**:
1. **Docs at root** (SELECTED):
   - Configuration: `routeBasePath: '/'` in docs preset
   - URLs: `/introduction`, `/quarter-overview`, etc.
   - Root URL (`/`) redirects to first doc (introduction)
2. Docs at `/docs/`:
   - Default Docusaurus behavior
   - URLs: `/docs/introduction`, `/docs/quarter-overview`
   - Root URL displays custom landing page (React component in `src/pages/index.tsx`)

**Decision**: Docs at root

**Rationale**:
- **Requirement**: FR-017 explicitly requires root URL redirect to introduction.md
- **User Experience**: Cleaner URLs for a documentation-only site (no blog, no custom landing page needed per Assumption A-019)
- **Simplicity**: Eliminates need to maintain separate landing page

**Trade-offs**:
- ✅ Meets specification requirement (FR-017)
- ✅ Simpler URL structure for users
- ❌ Cannot add custom homepage later without refactoring (acceptable for MVP per A-018)

**ADR Required**: ⚠️ **YES** - This decision impacts site architecture and URL structure. Document as ADR-002: "Root URL Configuration Strategy".

**Suggested ADR Title**: "ADR-002: Deploy Documentation at Root Path for Simplified Navigation"

### Decision 3: Content Extraction Methodology

**Options Considered**:
1. **Manual copy-paste** (SELECTED for MVP):
   - Manually identify section boundaries in source file
   - Copy each section to corresponding .md file
   - Add frontmatter manually
   - Verify formatting preservation by visual inspection
2. Automated script:
   - Write Node.js/Python script to detect headings (e.g., "## Quarter Overview")
   - Split file based on heading markers
   - Auto-inject frontmatter templates
   - Requires upfront scripting effort

**Decision**: Manual copy-paste (MVP approach per A-018)

**Rationale**:
- **Simplicity**: One-time operation, 10 files, script overhead not justified for MVP
- **Quality Control**: Manual review ensures 100% formatting preservation (FR-014)
- **Time-to-Delivery**: Faster for small-scale extraction (< 1 hour vs 1-2 hours scripting + debugging)
- **Assumption**: A-017 mentions "automated script" but also emphasizes MVP speed (A-018)

**Trade-offs**:
- ✅ Faster initial delivery
- ✅ Higher confidence in formatting preservation
- ❌ Not repeatable if source file changes (acceptable for stable textbook content)
- ❌ Human error risk (typos in frontmatter, incorrect section boundaries)

**ADR Required**: ⚠️ **MAYBE** - This is a tactical decision rather than architectural. If the textbook will be updated frequently, automation becomes architecturally significant. For a static textbook (likely assumption), no ADR needed.

**Recommendation**: Skip ADR for MVP. If source file requires re-extraction in future iteration, revisit and document as ADR-003: "Content Pipeline Automation Strategy".

**Options Considered**:

2. camelCase: `introduction.md`, `quarterOverview.md`, `whyPhysicalAiMatters.md`
3. PascalCase: `Introduction.md`, `QuarterOverview.md`, `WhyPhysicalAiMatters.md`

**Decision**: Kebab-case (lowercase with hyphens)

**Rationale**:
- **Docusaurus Best Practice**: Official documentation examples use kebab-case
- **URL Friendliness**: Produces clean URLs (`/why-physical-ai-matters` vs `/whyPhysicalAiMatters`)
- **Linux Compatibility**: Case-insensitive filesystems (Windows) vs case-sensitive (Linux deployment targets)

**Trade-offs**:
- ✅ Standard convention, widely recognized
- ✅ Better SEO (search engines favor hyphens over camelCase)
- ❌ None identified

**ADR Required**: ❌ **NO** - Standard industry practice, not architecturally significant.

## Appendix: Docusaurus Best Practices

### Configuration Best Practices

1. **TypeScript Config**: Use `import type` for type-only imports to avoid runtime overhead
2. **Sidebar Path**: Use relative path (`'./sidebars.ts'`) not `require.resolve()` in TypeScript configs
3. **Frontmatter Validation**: Docusaurus validates frontmatter schema automatically; invalid fields cause build errors

### Content Best Practices

1. **Heading Structure**: Each page should start with H1 (`#`) matching the frontmatter `title`
2. **Internal Links**: Use relative paths (`./other-page.md`) for cross-doc links
3. **Images**: Store in `static/` directory, reference as `/img/filename.png` in markdown

### Performance Best Practices

1. **Build Caching**: `.docusaurus/` directory caches builds; delete if stale
2. **Large Files**: Avoid committing `node_modules/` or `build/` to git (use `.gitignore`)
3. **Hot Reload**: Dev server watches file changes; save .md files to see updates instantly
